# modules/oidc

This module configures GitHub Actions CI/CD pipelines to authenticate with AWS without storing long-lived credentials in GitHub secrets. Every time an object is altered in the static website, CloudFront will invalidate that cached item so users always receive the latest version of updated objects — never stale cached ones.

Two roles are used — one to deploy Terraform with broad permissions, and another with minimal permissions so the pipeline can update the site and invalidate the CloudFront cache.

---

## How OIDC authentication works

Instead of storing AWS access keys in GitHub secrets, OIDC allows GitHub and AWS to trust each other directly through a short-lived identity token:

```
1. Push to main branch triggers GitHub Actions
        ↓
2. GitHub generates a short-lived OIDC token containing:
   {
     "aud": "sts.amazonaws.com",
     "sub": "repo:your-org/your-repo:ref:refs/heads/main"
   }
        ↓
3. Pipeline sends token to AWS STS asking to assume the role
        ↓
4. AWS verifies the token against the two conditions:
   - aud == "sts.amazonaws.com"?            ✓
   - sub matches your repo on main branch?  ✓
        ↓
5. AWS issues temporary credentials (expire after job finishes)
        ↓
6. Pipeline uses those credentials to run Terraform or deploy site
```

The thumbprint (`6938fd4d98bab03faadb97b34396831e3780aea1`) is a fingerprint of GitHub's SSL certificate — it is GitHub's identity, not yours. It is the same value for every AWS account that integrates with GitHub Actions and is safe to commit publicly.

## What this module creates

### 1. OIDC Provider

Registers GitHub as a trusted identity provider in your AWS account. This is the foundation — without it, AWS would not know to trust tokens coming from GitHub.

---

### 2. Shared Trust Policy

Both roles use the same trust policy — only your specific repo on the main branch can assume them. Two conditions enforce this:

**Condition 1 — `aud` (audience):** Verifies the token was intended for AWS STS, not another service.

**Condition 2 — `sub` (subject):** Verifies the token came from your specific repo on the main branch — not any other repo or branch on GitHub. Even another repo in your own GitHub account could not assume these roles.

```
Both conditions must pass (AND logic) before AWS issues credentials.
```

---

### 3. Role 1 — Terraform (`github-actions-terraform`)

Broad permissions scoped to managing this project's infrastructure only.

#### State bucket — fully scoped

The pipeline needs to read, write, and delete the state file, and list the bucket to check if it exists. Scoped to the exact state file key — not the entire bucket:

```
s3:GetObject, s3:PutObject, s3:DeleteObject → scoped to terraform.tfstate file only
s3:ListBucket                               → scoped to state bucket only
```

#### DynamoDB lock — fully scoped

Terraform uses DynamoDB to prevent two pipelines from running `apply` at the same time. Scoped to the exact lock table using the account ID from `data.aws_caller_identity` — no wildcards on account:

```
dynamodb:GetItem, dynamodb:PutItem, dynamodb:DeleteItem → lock table only
```

#### Site S3 bucket — scoped to specific bucket

Terraform needs full S3 access to manage the static site bucket and its configuration:

```
s3:* → scoped to:
  var.static_website_bucket_arn      (e.g. arn:aws:s3:::leomarques-portfolio-site)
  var.static_website_bucket_arn/*    (e.g. arn:aws:s3:::leomarques-portfolio-site/*)
```

The bucket ARN without `/*` covers bucket-level operations such as getting the bucket configuration. The `/*` covers reading and writing individual objects inside it. Together they give Terraform full access to this bucket and nothing else.


#### CloudFront — scoped to specific distribution

Scoped to this project's distribution only — not all CloudFront distributions in the account:

```
cloudfront:CreateInvalidation → this distribution only
```

#### IAM — scoped to roles and policies only, no users or groups

IAM must remain broad enough for Terraform to create roles dynamically, but restricted to roles and policies only — never users, groups, or account-level settings. Further scoped to the project naming convention so only resources prefixed with `portfolio-` or `github-actions-` can be managed:

```
iam:* → roles/portfolio-*, roles/github-actions-*, policies/portfolio-*, oidc-provider/*
```

#### API Gateway and CloudWatch Logs — must remain broad

API Gateway resource IDs are generated by AWS and are not predictable before the first apply. CloudWatch log group ARNs are created at Lambda runtime, not at apply time. These two services cannot be scoped to specific ARNs without breaking the first deployment:

```
apigateway:* → *
logs:*        → *
```

---

### 4. Role 2 — Deploy (`github-actions-deploy`)

Minimal permissions — only what the pipeline needs to upload files to the static site bucket and invalidate the CloudFront cache. This role runs on every push to main. If it were ever compromised, an attacker could only touch site files — not infrastructure.

```
s3:ListBucket          → site bucket only (checks what files exist before syncing)
s3:PutObject           → site bucket objects only (upload new or updated files)
s3:DeleteObject        → site bucket objects only (remove files deleted from repo)
cloudfront:CreateInvalidation → this distribution only (purge stale cached files)
```

## Permission comparison

| Permission | Terraform role | Deploy role |
|---|---|---|
| State bucket read/write | ✓ scoped to file | ✗ |
| DynamoDB lock | ✓ scoped to table | ✗ |
| Site S3 full access | ✓ scoped to bucket | ✗ |
| Site S3 upload/delete | ✓ | ✓ scoped to bucket |
| CloudFront invalidation | ✓ scoped to distribution | ✓ scoped to distribution |
| IAM roles and policies | ✓ scoped to naming convention | ✗ |
| API Gateway | ✓ broad | ✗ |
| CloudWatch Logs | ✓ broad | ✗ |
| IAM users/groups | ✗ | ✗ |

## Inputs

| Variable | Type | Description |
|---|---|---|
| `github_org` | `string` | GitHub username |
| `github_repo` | `string` | Repository name |
| `tfstate_bucket` | `string` | Name of the S3 bucket storing Terraform state |
| `tf_state_bucket_key` | `string` | Folder path for state file e.g. `portfolio-website` |
| `dynamodb_lock_table` | `string` | Name of the DynamoDB lock table |
| `dynamodb_lock_table_region` | `string` | AWS region where the DynamoDB lock table lives |
| `static_website_bucket_arn` | `string` | ARN of the static website S3 bucket |
| `cloudfront_distribution_arn` | `string` | ARN of the CloudFront distribution |
| `aws_region` | `string` | AWS region where infrastructure is deployed |
| `tags` | `map(string)` | Tags from locals in `infra/main.tf` |

---

## Outputs

| Output | Sensitive | Used by |
|---|---|---|
| `terraform_role_arn` | yes | GitHub secret `TERRAFORM_ROLE_ARN` → deploy.yml Terraform job |
| `deploy_role_arn` | yes | GitHub secret `DEPLOY_ROLE_ARN` → deploy.yml deploy job |

Retrieve after `terraform apply`:

```bash
terraform output -raw terraform_role_arn
terraform output -raw deploy_role_arn
```

---

## GitHub secrets required

After `terraform apply`, add these four secrets to your GitHub repo under Settings → Secrets and variables → Actions:

```
TERRAFORM_ROLE_ARN           ← terraform output -raw terraform_role_arn
DEPLOY_ROLE_ARN              ← terraform output -raw deploy_role_arn
BUCKET_NAME                  ← terraform output -raw s3_bucket
CLOUDFRONT_DISTRIBUTION_ID   ← terraform output -raw cloudfront_distribution_id
```

## CI/CD pipeline flow

```
Push to main
    ↓
┌─────────────────────────────────┐
│  Job 1: terraform               │
│  assumes github-actions-terraform│
│                                 │
│  terraform init                 │
│  terraform plan                 │
│  terraform apply                │
└─────────────────┬───────────────┘
                  │ needs: terraform (waits for job 1)
┌─────────────────▼───────────────┐
│  Job 2: deploy                  │
│  assumes github-actions-deploy  │
│                                 │
│  aws s3 sync site/ → S3         │
│  cloudfront create-invalidation │
└─────────────────────────────────┘
```

Job 2 only runs after Job 1 succeeds — site files are never deployed on top of a failed infrastructure apply.
